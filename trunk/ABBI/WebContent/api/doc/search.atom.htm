Search Results Output

Search results generated by this application can be produced in a variety of formats as specified by:

  • IHE's Mobile Access to Health Documents Profile
  • HL7's Fast Health Information Resources DocumentHeader Resource
  • ABBI's Atom Format

Each of these formats uses the Atom Syndication Format to wrap the response.

ABBI Feeds

An ABBI feed starts with a top level <feed> tag in application/atom+xml format. The <feed> element will contain the following elements:

<id>

This must be a unique id for the feed for the patient. The feed for the demonstration application provides the base URL for the search API call which generated this feed, and includes the patientId query parameters in the URL to differentiate between multiple feeds from the same site for different patients. No other query parameters will be included in the URL. The value of patientId query parameter will be URL encoded.

<generator>

The feed will contain a <generator> element describing the application used to generate the content. The URI attribute should point to a web site describing where the content generator is described. The version attribute should be present to indicate what version of the application was used to generate the content. The text of this element should provide a short text description of the generator.

<link>

The feed will have at least one link[@rel="self"] containing information about the API call that generated the content. That link contains the full URL of the search API call which produced the content in the href attribute. The rel attribute will be equal to self. No other <link> element may be present with rel="self", but other links may be present.

<icon>

The feed may provide an icon. Feeds generated by the demonstration application will generate a URL to this icon image. See the credits page for more details on this image.

<logo>

The feed may provide a logo.  Feeds generated by the demonstration application will generate a URL to this logo image. See the credits page for more details on this image.

<rights>

The feed should provide text describing application rights and restrictions on the content as agreed to and authorized by the patient or their authorized representative. An example of what might appear can be found below.

Content in this feed is provided by (Sending Organization) to receiving applications as authorized by the patient or authorized representative. The information contained herein may contain sensitive patient information, individually identifiable data or other protected health information, and   should be protected by adequate security mechanism. The provider of this service communicates this information securely.  Responsibility for securing this information after it has been delivered through this service is left to the receiving application, and/or the patient or their authorized representative.

<contributor>

The feed may describe contributors (e.g., organizations that provided content) to content in the feed.

<title>

The feed will provide a title describing the content in general terms. The title should indicate who this information pertains to.

<subtitle>

The feed may provide a subtitle describing the content in more detail. The sample application describes the content in terms of what was queried.

<category>

The feed should report the categories associated with the search query terms used to generate the content (e.g., format, class and mimeType).

<updated>

The feed will provide the date which the content is generated in the <updated> tag.

<author>

The feed SHOULD NOT usually contain <author> elements.  Each document in the feed must have an individually identified author.  Feeds which generate content from a single author (e.g., patient generated content) may include an <author> element, but <author> elements must be present on each document.  The demonstration application does not generate <author> elements in feeds.
